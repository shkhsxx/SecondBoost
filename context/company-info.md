# 세컨부스트 도메인 컨텍스트

## 기업 소개

**세컨부스트(SecondBoost)** 는 CTO 없는 초기 스타트업의 제품 개발을 대행·가속하는 **MVP 빌드 스튜디오**입니다.

- **슬로건**: "단 10주 만에 아이디어를 AI SaaS 제품으로 전환"
- **핵심 미션**: 기술 인력이 부족한 초기 창업자가 검증 가능한 제품(MVP)을 빠르게 손에 쥐게 한다
- **차별점**: NLP·멀티모달·음성 AI 등 최신 AI 파이프라인을 사내에 내재화 — 외주가 아닌 "구원투수형 제품팀"
- **CEO**: 조바울 (10년 경력 개발자 출신)
- **주요 사업**: MVP 개발 · AI SaaS 빌드 · 기술 컨설팅

## 주요 서비스

| 서비스 | 설명 |
|--------|------|
| MVP 빌드 (10주 스프린트) | 아이디어·요구사항 → 동작하는 SaaS MVP까지 10주 내 구현 |
| AI SaaS 빌드 | NLP·멀티모달·음성 AI 파이프라인을 제품에 내재화 |
| 기술 컨설팅 | CTO 부재 스타트업에 아키텍처·보안·확장성 자문 |
| 요구사항 → PRD 자동화 | 슬랙·이메일·회의록 비정형 텍스트를 표준 PRD로 자동 변환 (본 과제 영역) |

## 고객 세그먼트

| 세그먼트 | 사례 | 주요 니즈 |
|----------|------|-----------|
| 초기 스타트업 창업자 | 비개발자 출신 1인 창업자 | 아이디어를 빠르게 검증할 MVP, 기술 의사결정 대행 |
| 제품 기획자 | 사내 PM·기획자 | 흩어진 요구사항을 개발 가능한 스펙으로 정리 |
| 시드~프리A 투자유치 팀 | 데모데이·투자자 미팅 준비 팀 | 데모 가능한 제품을 마감 일정 내 완성 |
| 비기술 도메인 전문가 | 헬스케어·커머스·핀테크 현업 전문가 | 도메인 아이디어의 제품화, 기술 번역 |

## 왜 이 과제(PRD 자동 정리)가 중요한가 (비즈니스 맥락)

세컨부스트의 PM은 고객사로부터 요구사항을 **슬랙 DM·카카오톡·CS 이메일·구두 회의록** 등 비정형 채널로 받습니다. 이 흩어진 텍스트를 표준 PRD로 변환하는 작업이 매 프로젝트 반복됩니다.

**PRD 자동 정리가 비즈니스에 중요한 이유:**

1. **10주 일정의 첫 병목 제거**: 케이스당 2~4시간 걸리는 수작업 구조화를 단축 → 개발 착수 시점이 앞당겨짐
2. **재작업 비용 차단**: 기능 정의가 불명확하면 후반 스펙 변경 → 일정·비용 폭증. 초기에 [TODO]로 잡아내면 예방
3. **모순 조기 감지**: 창업자·기획자·CS·보안 등 여러 출처의 상반된 요구를 PM이 놓치면 출시 직전 대형 변경 발생
4. **의사결정 가시화**: "일단/나중에/논의 중"으로 떠다니던 미결 항목을 [TODO]로 명시화 → 고객과의 합의 속도 향상
5. **PM 역량 표준화**: 신입 PM도 동일 품질의 PRD를 생산 → 스튜디오 전체 처리량 증가

## PRD란 무엇인가 (표준 6섹션)

**PRD(Product Requirements Document)** 는 제품의 목적·기능·동작 방식·제약을 정리한 "단일 진실의 원천(Single Source of Truth)"입니다. 개발·디자인·QA가 같은 기준으로 제품을 만들게 합니다.

```
1. 배경 (Background)        — 왜 필요한가, 어떤 문제를 푸는가
2. 목표 (Goals)            — 달성 목표 + 성공 지표 + Out-of-scope
3. 기능 정의 (Feature)      — 기능 목록 + 동작(입력→처리→출력) + 우선순위(P1/P2/P3 또는 MoSCoW)
4. 사용자 시나리오 (Scenario) — 페르소나 + Happy path + Sad path
5. 예외 케이스 (Edge Case)   — 경계 조건·빈 값·오류 처리·롤백
6. TODO / 미결 사항         — 확인 필요·결정 대기·다음 스프린트 이월
```

> 본 과제의 출력은 케이스당 **5섹션 압축 포맷**(기능 정의 / 사용자 시나리오 / 우선순위 / 예외 케이스 / 미결 항목)을 사용합니다. output/template.md 참고.

## 비정형 → 정형 변환의 핵심 원칙

### 비정형 텍스트의 특징 (슬랙/이메일/메모)
- **발화 순서가 뒤섞임**: 요구사항이 시간 순으로 흩어져 구조화 안 됨
- **맥락 의존적**: "그거", "저번에 말한 것처럼" 같은 지시어
- **감정·의견 혼재**: "있으면 좋고 없어도 되고 ㅋㅋ" 같은 약한 요구
- **모순 발생**: 다른 관계자(또는 같은 사람)가 상충되는 요구를 동시/번복
- **결정 미완성**: "일단", "나중에", "논의 중" 등 확정되지 않은 사항

### 변환 시 핵심 원칙
1. **사실과 의견 분리**: "이렇게 하면 좋겠다"(의견) vs "이렇게 해야 한다"(요구)
2. **모순 명시화**: 충돌하는 요구는 본문에 단정하지 말고 [TODO]로 의사결정 요청
3. **불명확 항목 [TODO]화**: 추측 금지. 원문에 없는 정책·수치는 본문이 아니라 [TODO]
4. **관계자별 요구 분리**: 누가 요청했는지(창업자/기획자/CS/보안 등) 출처 추적

## 우선순위 판단 기준 (본 과제 권장)

| 우선순위 | 의미 | 원문 신호 |
|----------|------|-----------|
| **P1** | 즉시 (이번 스프린트) | "꼭", "반드시", "이번 스프린트", "데모 다음주", 핵심 흐름 차단 |
| **P2** | 다음 스프린트 | "필요할 것 같은데", 보조 기능, 합리적이나 급하지 않음 |
| **P3** | 백로그 | "나중에", "있으면 좋고", "다음에~", 부가 편의 |

## 참고 용어 사전

| 용어 | 정의 |
|------|------|
| **PRD** | Product Requirements Document. 제품 요구사항 정의서 |
| **[TODO]** | 현재 확인 불가·결정 대기 항목 표시. 본 과제의 핵심 산출물. "[TODO] {누구}에게 {무엇} 확인" 형태 권장 |
| **사용자 시나리오** | 특정 사용자가 제품을 쓰는 흐름. 본 과제는 1.트리거 → 2.행동 → 3.결과 3단계로 서술 |
| **예외 케이스(Edge Case)** | 빈 값·최대값·동시 접근 등 비일반적 입력·상황에서의 동작 조건 |
| **우선순위(Priority)** | P1(즉시) / P2(다음 스프린트) / P3(백로그). 또는 MoSCoW |
| **MoSCoW** | Must / Should / Could / Won't have — 우선순위 프레임워크 |
| **Happy Path** | 사용자가 오류 없이 목표를 달성하는 정상 시나리오 |
| **Sad Path** | 오류 발생·목표 미달성 시나리오 |
| **Out-of-scope** | 이번 PRD 범위에서 제외되는 기능·요구 |
| **Single Source of Truth** | 팀 전체가 참조하는 단일 공식 문서 |
| **스프린트(Sprint)** | 보통 1~2주 단위의 개발 주기 |
| **관계자(Stakeholder)** | 제품에 이해관계가 있는 모든 사람 (창업자·PM·개발자·디자이너·CS·보안 등) |
| **모순(Conflict)** | 동일 기능에 대한 상반된 요구. 감지 후 [TODO]로 의사결정 요청 |
