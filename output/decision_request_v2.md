## 🔴 Blocker — 개발 착수 전 반드시 결정

### B-01. 리뷰 신고 후 숨김 처리 기준
**관련 케이스:** C03 (Conflict)
**임팩트:** 숨김 기준이 결정되어야 신고 처리 로직 개발 가능. 자동 숨김 vs 운영자 검토 방식은 구현 방식이 완전히 다름
**긴급도:** 최우선 — Conflict 미결 상태에서 해당 기능 개발 착수 불가

| 구분 | 내용 |
|------|------|
| 요구 A | 기획자 + 창업자 (슬랙): 신고 3건 이상 시 자동 숨김 처리 |
| 요구 B | 운영팀 (주간 회의): 신고 1건이어도 운영자 검토 후 처리. 자동 숨김은 허위 신고 악용 가능성 |
| 미결 시 리스크 | 개발팀이 임의 선택 시 출시 후 전면 재설계 가능성 |
| 결정 요청 | **PO가 자동 숨김 기준 vs 운영자 검토 방식 최종 결정** |

## 🟡 주의 — 스프린트 내 결정 권장

### W-01. 업그레이드 차액 정산 방식
**관련 케이스:** C02
**임팩트:** 차액 정산 방식이 결정되어야 업그레이드 결제 처리 로직 개발 가능
**결정 요청:** 내부 논의 완료 후 운영팀장이 PM에게 공유 (원문: "아직 결정이 안 됐습니다")

---

### W-02. 최근 본 상품 최대 표시 개수
**관련 케이스:** C01
**임팩트:** 개수 미결 시 표시 로직 상한선 설계 불가
**결정 요청:** 창업자에게 최대 표시 개수 결정 요청 (원문: "10개? 20개? 잘 모르겠어요 일단 적당히")
