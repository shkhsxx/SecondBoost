# SecondBoost PRD 파이프라인 — Standard 레벨 제출

---

## 1. 재사용 가능한 파이프라인

### 슬래시 커맨드

```
/prd [파일경로]
```

**예시**
```
/prd data/raw_requirements.txt
/prd data/raw_requirements_v2.txt
```

**위치:** `.claude/commands/prd.md`
**조건:** 입력 파일이 아래 구분자 형식을 따르면 어떤 파일이든 동일하게 동작한다.

```
--- [케이스 N · 유형] ---
{비정형 텍스트}

--- [케이스 N+1 · 유형] ---
{비정형 텍스트}
```

지원 유형: `슬랙 DM` / `슬랙 채널 스레드` / `CS 이메일` / `구두 회의 메모` / `복합`

### 파이프라인 흐름

```
입력 파일
    │
    ▼
[Step 1] 파일 읽기 & 케이스 분리
    │
    ▼
[Step 2] 요구사항 구조화
         · 확정 vs 의견 분리
         · TODO 탐지
         · Conflict 탐지
         · 우선순위 판단
    │
    ▼
[Step 3] PRD 문서 생성
         · 케이스별 5섹션 PRD
         · QA 체크포인트
         · 개발 참고사항
    │
    ▼
[Step 4] 출시 전 의사결정 요청서 생성
         · Blocker / 주의 / 참고 분류
    │
    ▼
[Step 5] 파일 저장 (output/ 폴더)
         decisions_[파일명].md
         prd_[파일명].md
         decision_request_[파일명].md
```

### 재현 검증

`raw_requirements_v2.txt` (3케이스 신규 입력)으로 파이프라인을 재실행하여 동일 품질의 출력물 생성 확인:
- `output/decisions_v2.md`
- `output/prd_v2.md`
- `output/decision_request_v2.md`

## 2. 파이프라인으로 생성한 PRD 샘플

**파일:** `output/prd_submission.md`
**원문:** `data/raw_requirements.txt` (15개 케이스)

포함 내용:
- 15개 케이스 전체 5섹션 PRD
- 복합 케이스 2건(C14·C15) 모순 양쪽 병기 후 [TODO] 처리
- 불명확 항목 누락 없이 [TODO] 표시

## 3. PRD 설계 근거 — "개발·QA에 무엇이 유용한가"를 기준으로

### 3-1. 섹션 구성과 설계 이유

| 섹션 | 설계 이유 |
|------|-----------|
| **기능 정의** (1줄) | 개발자가 기능 범위를 오해 없이 파악하도록. 한 줄 제한으로 범위 크리프 방지 |
| **사용자 시나리오** | QA 테스트케이스의 기반. 시나리오 없으면 QA가 경계값을 누락함. Happy path + 예외 시나리오를 함께 제공 |
| **우선순위 (원문 근거 포함)** | 근거 없는 우선순위는 신뢰받지 못함. 원문 표현을 그대로 인용해 PM이 임의로 상향/하향하지 않았음을 증명 |
| **예외 케이스** | QA가 엣지케이스를 놓치지 않도록. 미결 예외는 [TODO]로 표시해 "정책 확정 후 테스트" 범위를 명확히 함 |
| **미결 항목 (TODO)** | 개발 착수 전 해소해야 할 항목 추적. 삭제 금지 — 미결이 사라지면 개발 중에 발견됨 |

**섹션을 5개로 압축한 이유:**
PRD 섹션이 너무 많으면 작성자도 독자도 피로해진다. "개발이 시작하려면 무엇이 필요한가"로 역산했을 때 필수 정보가 5개 섹션에 수렴했다. 정책·입출력·REQ 번호 등 상세 정보는 필요 시 별도 확장 가능하다.

---

### 3-2. 우선순위 체계 (P1 / P2 / P3)

| 등급 | 트리거 | 의미 | 대응 |
|------|--------|------|------|
| **P1** | "반드시", "필수", "꼭", "긴급", "즉시", 데모/마감 언급 | 이번 스프린트 내 구현 | 즉시 착수 |
| **P2** | "권장", "있으면 좋음", "VoC 많음", 다음 스프린트 | 가까운 시일 내 구현 | 다음 스프린트 계획 |
| **P3** | "나중에", "추후", "이번엔 제외", "백로그" | 확정되지 않았거나 이번 배포 외 | 백로그 등록 |

**이 체계를 선택한 이유:**
- 수치 기반(1~5점)보다 원문 언어 기반 분류가 PM이 자의적으로 조작할 여지를 줄인다.
- 3단계로 단순화한 이유: 4~5단계는 경계가 모호해져 등급 판단에 시간이 더 소요됨. 3단계면 "지금 / 곧 / 나중에"로 직관적으로 판단 가능.
- **판단 원칙:** 근거가 없으면 P2로 기본 분류하고 TODO를 생성한다. PM이 임의로 P1을 부여하지 않는다.

### 3-3. [TODO] 표기 규칙

**생성 기준 — 아래 표현이 원문에 있으면 반드시 TODO를 생성한다**

| 원문 표현 | 의미 |
|-----------|------|
| "미정" | 정책/수치가 결정되지 않음 |
| "논의 중" / "얘기해요" | 이해관계자 간 합의가 필요 |
| "일단" | 임시 결정, 추후 변경 가능성 있음 |
| "알아서" | 담당자에게 위임된 결정 |
| "나중에" / "추후" | 다음 시점으로 미뤄진 사항 |
| "모르겠어요" / "고민" | 결정 주체가 불명확 |

**작성 형식**
```
[TODO] {담당자}에게 {무엇을} 확인/결정 요청
```

담당자 우선순위: 원문에서 언급된 인물 → Product Owner → PM

**처리 원칙:**
- **절대 삭제하지 않는다.** 미결 사항이 사라지면 개발 중에 발견된다.
- TODO가 해소된 경우: 담당자가 직접 파일에서 TODO를 확정 요건으로 변경한다.
- Blocker TODO는 개발 착수 전 반드시 해소한다.

**이 규칙을 설계한 이유:**
원문에서 미결 표현을 키워드 기반으로 탐지하면 PM의 판단이 개입할 여지가 줄어든다. "모르겠어요"나 "일단"처럼 구어체 표현도 커버하도록 목록을 구성했다. TODO를 삭제 금지로 명시한 이유는, 현장에서 미결 사항을 "일단 넘기고" 개발 중에 발견하는 패턴이 반복되기 때문이다.

---

### 3-4. Conflict 처리 규칙

**감지 기준 — 아래 중 하나라도 해당하면 Conflict로 처리**

| 조건 | 예시 |
|------|------|
| 같은 기능에 상반된 요구가 두 곳 이상에서 등장 | 창업자: 매직링크 / 보안 컨설턴트: 비밀번호+OTP |
| 한 사람이 대화 중 번복 | 창업자: 서버 저장 → 로컬 저장으로 번복 |
| 채널(슬랙 vs 이메일 vs 회의)에 따라 상반된 결론 | 슬랙 "3건 자동 숨김" vs 회의 "운영자 검토" |

**작성 형식**
```
[CONFLICT] {기능명}
- 요구 A ({출처}): {원문 내용}
- 요구 B ({출처}): {원문 내용}
- [TODO] Product Owner에게 최종 정책 결정 요청
```

**이 규칙을 설계한 이유:**
PM이 Conflict를 임의로 해소하면 책임 소재가 불명확해진다. 양쪽을 원문 그대로 기록하고 PO에게 결정을 위임하는 것이 맞다. Conflict 상태에서 개발을 시작하면 나중에 전면 재설계가 필요해지므로, Conflict가 있는 케이스는 Blocker로 분류한다.
