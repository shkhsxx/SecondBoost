# PRD 파이프라인 설계 명세서

> SecondBoost 프로젝트 — Standard 레벨 파이프라인 설계 문서
> 대상 독자: PM, 개발팀, QA

---

## 1. 파이프라인 개요

### 목적

비정형 요구사항(슬랙 DM, CS 이메일, 구두 회의 메모 등)을 입력받아 개발팀과 QA가 바로 사용할 수 있는 표준 PRD를 자동 생성한다.

### 실행 방법

Claude Code에서 아래 한 줄로 전체 파이프라인을 실행한다.

```
/prd [raw_requirements.txt 경로]
```

예시:

```
/prd data/raw_requirements.txt
```

### 파이프라인 흐름

```
raw_requirements.txt
        │
        ▼
  [Step 1] 파일 읽기 & 케이스 분리
        │
        ▼
  [Step 2] 요구사항 구조화 (decisions 문서)
   - 사실/의견 분리
   - TODO 탐지
   - Conflict 탐지
   - 우선순위 판단
        │
        ▼
  [Step 3] PRD 문서 생성
   - 케이스별 PRD 섹션 작성
   - QA 체크포인트 생성
   - 개발 참고사항 추가
        │
        ▼
  [Step 4] 파일 저장
   decisions_[파일명].md
   prd_[파일명].md
```

### 출력물

| 파일                      | 용도                        |
| ------------------------- | --------------------------- |
| `decisions_[파일명].md` | PM 검토용 중간 분석 문서    |
| `prd_[파일명].md`       | 개발팀 및 QA 최종 참고 문서 |

## 2. PRD 섹션 설계

### 섹션 구성 및 설계 근거

| 섹션                          | 내용                          | 설계 근거                                                                              |
| ----------------------------- | ----------------------------- | -------------------------------------------------------------------------------------- |
| **기능 정의**           | 기능을 한 문장으로 요약       | 개발자가 기능의 범위를 오해 없이 파악하도록. 한 문장 제한으로 범위 크리프 방지         |
| **기능 목적**           | 왜 만드는가                   | 개발 중 의사결정 시 근거로 활용. "왜"를 모르면 구현 방향이 틀어짐                      |
| **사용자 시나리오**     | 행동 흐름 (Happy path + 예외) | QA 테스트케이스의 기반. 시나리오가 없으면 QA가 경계값을 누락함                         |
| **기능 요구사항 (REQ)** | 번호가 붙은 구체적 요건       | 개발 티켓 단위와 1:1 매핑 가능하도록 번호 부여. 추적성 확보                            |
| **정책**                | 로그인/권한/저장/알림/보안 등 | 정책은 여러 기능에 걸쳐 영향을 주므로 별도 섹션으로 분리하여 일관성 유지               |
| **입력값 / 출력값**     | 시스템 경계 명확화            | API 설계 및 유효성 검증 범위를 명확히 하기 위함                                        |
| **예외 케이스**         | 예외 상황 + 처리 방식 + TODO  | QA가 엣지케이스를 놓치지 않도록. TODO 컬럼으로 미결 예외 추적                          |
| **우선순위**            | P1/P2/P3 + 원문 근거          | 근거 없는 우선순위는 신뢰받지 못함. 반드시 원문 표현을 근거로 명시                     |
| **개발 참고사항**       | 기술적 고려사항               | 개발자가 설계 착수 전에 알아야 할 제약, 의존성, 기술 검토 사항                         |
| **QA 체크포인트**       | 체크리스트 형식 테스트 항목   | QA가 별도 테스트케이스를 작성하지 않아도 기본 검증이 가능하도록                        |
| **Conflict**            | 상반된 요구 원문 유지         | Conflict를 PM이 임의 해소하면 책임 소재가 불명확해짐. 반드시 원문 유지 후 PO 결정 요청 |
| **미결 항목(TODO)**     | 결정되지 않은 모든 항목       | 개발 착수 전 해소해야 할 항목 추적. 삭제 금지                                          |

## 3. 우선순위 체계

### 기준표

| 등급         | 트리거 키워드                                          | 의미                           | 대응               |
| ------------ | ------------------------------------------------------ | ------------------------------ | ------------------ |
| **P1** | "반드시", "필수", "꼭", "긴급", "즉시", 데모/마감 언급 | 현재 스프린트 내 구현 필요     | 즉시 개발 착수     |
| **P2** | "권장", "있으면 좋음", "VoC 많음", 다음 스프린트       | 가까운 시일 내 구현 권장       | 다음 스프린트 계획 |
| **P3** | "추후", "나중에", "이번엔 제외", "백로그"              | 확정되지 않았거나 이번 배포 외 | 백로그 등록        |

### 판단 원칙

- 반드시 **원문 표현**을 근거로 우선순위를 결정한다.
- PM이 임의로 우선순위를 상향/하향하지 않는다.
- 근거가 없으면 P2로 기본 분류하고 TODO를 생성한다.

## 4. TODO 표기 규칙

### 생성 기준

아래 표현이 원문에 있으면 반드시 TODO를 생성한다.

| 원문 표현              | 의미                             |
| ---------------------- | -------------------------------- |
| "미정"                 | 정책/수치가 결정되지 않음        |
| "논의 중" / "얘기해요" | 이해관계자 간 합의가 필요        |
| "일단"                 | 임시 결정, 추후 변경 가능성 있음 |
| "알아서"               | 담당자에게 위임된 결정           |
| "나중에" / "추후"      | 다음 시점으로 미뤄진 사항        |
| "모르겠어요" / "고민"  | 결정 주체가 불명확               |

### 작성 형식

```
[TODO] {담당자}에게 {무엇을} 확인/결정 요청
```

담당자 우선순위: 원문에서 언급된 인물 → Product Owner → PM

### TODO 처리 원칙

- **절대 삭제하지 않는다** — 미결 사항이 사라지면 개발 중 발견됨
- TODO가 해소된 경우: 담당자가 직접 파일에서 TODO를 확정 요건으로 변경
- Blocker TODO는 개발 착수 전 반드시 해소

## 5. Conflict 감지 규칙

### 감지 기준

동일 기능에 대해 아래 조건 중 하나라도 해당하면 Conflict로 처리한다.

| 조건                                            | 예시                                           |
| ----------------------------------------------- | ---------------------------------------------- |
| 같은 기능에 상반된 요구가 두 곳 이상에서 등장   | 창업자: 매직링크 / 보안 컨설턴트: 비밀번호+OTP |
| 한 사람이 대화 중 번복                          | 창업자: 서버 저장 → 로컬 저장으로 번복        |
| 채널(슬랙 vs 이메일 vs 회의)에 따라 상반된 결론 | —                                             |

### 처리 원칙

- PM이 임의로 하나를 선택하지 않는다.
- 두 요구를 모두 원문 표현으로 기록한다.
- 반드시 Product Owner에게 최종 결정을 요청한다.

### 작성 형식

```
[CONFLICT] {기능명}
- 요구 A ({출처}): {원문 내용}
- 요구 B ({출처}): {원문 내용}
- [TODO] Product Owner에게 최종 정책 결정 요청
```

## 6. 의사결정 우선순위 분류 기준

PRD 문서 마지막의 "출시 전 의사결정 필요 사항"은 아래 기준으로 분류한다.

| 등급              | 기준                                             | 대응             |
| ----------------- | ------------------------------------------------ | ---------------- |
| **Blocker** | 미결 시 개발 착수 불가 (구현 방식이 결정에 종속) | 즉시 PO 미팅     |
| **High**    | 미결 시 일정 지연 발생 가능                      | 스프린트 내 결정 |
| **Medium**  | 미결 시 QA 테스트 범위 불명확                    | QA 착수 전 결정  |
| **Low**     | 출시 후 대응 가능                                | 백로그 등록      |

## 7. 개발/QA 관점 설계 근거

### 개발 관점

| 설계 선택                       | 근거                                                                                        |
| ------------------------------- | ------------------------------------------------------------------------------------------- |
| REQ에 번호 부여 (REQ-001)       | 개발 티켓과 1:1 추적 가능. "어떤 요건의 구현인가"를 코드 리뷰 시 명확히 참조 가능           |
| 정책을 별도 섹션으로 분리       | 정책은 여러 기능에 걸쳐 영향을 줌. 한 곳에서 변경 시 전체 일관성 유지 가능                  |
| Conflict 미결 시 개발 착수 금지 | Conflict 상태에서 개발하면 나중에 전면 재설계가 필요해짐. 선결 비용보다 후불 비용이 훨씬 큼 |
| 입력값/출력값 명시              | API 설계, 유효성 검증 범위, 에러 응답 설계의 기준이 됨                                      |

### QA 관점

| 설계 선택                            | 근거                                                                         |
| ------------------------------------ | ---------------------------------------------------------------------------- |
| QA 체크포인트를 체크리스트로         | QA 담당자가 테스트케이스를 처음부터 작성하지 않아도 기본 검증 항목 확보      |
| 예외 케이스에 TODO 컬럼              | 미결 예외의 테스트 항목을 "정책 확정 후 테스트"로 명시하여 QA 범위 오해 방지 |
| 사용자 시나리오에 예외 시나리오 포함 | Happy path만 테스트하는 QA 누락을 방지. 예외 흐름도 시나리오로 제공          |
| 우선순위에 원문 근거 포함            | QA 리소스 배분 시 P1 기능에 집중할 객관적 근거 제공                          |

## 8. 슬래시 커맨드 파일 위치

```
[06] 세컨부스트/
├── .claude/
│   └── commands/
│       └── prd.md          ← /prd 커맨드 정의 파일
├── data/
│   └── raw_requirements.txt
└── output/
    ├── decisions_[파일명].md
    ├── prd_[파일명].md
    └── prd_design_spec.md  ← 이 문서
```

## 9. 입력 파일 형식 가이드

파이프라인이 정상 동작하려면 입력 파일이 아래 형식을 따라야 한다.

```
--- [케이스 N · 유형] ---
{비정형 텍스트}

--- [케이스 N+1 · 유형] ---
{비정형 텍스트}
```

지원 유형: `슬랙 DM`, `슬랙 채널 스레드`, `CS 이메일`, `구두 회의 메모`, `복합`

케이스 구분자가 없으면 파이프라인이 케이스를 자동으로 분리하지 못한다.
