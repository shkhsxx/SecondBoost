## C01 · 최근 본 상품 목록 표시

### 기능 정의

상품 상세 페이지에서 유저가 최근에 조회한 상품 목록을 표시하는 기능

### 기능 목적

유저가 이전에 조회한 상품을 빠르게 재접근할 수 있도록 함

### 사용자 시나리오

**Happy Path (로그인 유저)**
1. 유저가 상품 상세 페이지를 조회함
2. 최근 본 상품 목록에 해당 상품이 추가됨 (서버 저장)
3. 유저가 다른 상품을 조회해도 이전 상품이 목록에 유지됨

**Happy Path (비로그인 유저)**
1. 비로그인 유저가 상품 상세 페이지를 조회함
2. 최근 본 상품 목록에 해당 상품이 추가됨 (브라우저 저장)
3. 브라우저를 닫아도 다음 접속 시 목록이 유지됨

**예외 시나리오**
- 비로그인 상태로 조회 후 로그인 전환 시 브라우저 저장 목록 처리 — 정책 미결 (TODO)

### 기능 요구사항 (REQ)

| ID | 요구사항 | 출처 |
|----|---------|------|
| REQ-001 | 상품 상세 페이지에 최근 본 상품 목록을 표시한다 | 창업자 (슬랙 DM) |
| REQ-002 | 비로그인 유저의 최근 본 상품은 브라우저에 저장한다 | 창업자 (슬랙 DM) |
| REQ-003 | 로그인 유저의 최근 본 상품은 서버에 저장한다 | 창업자 (슬랙 DM) |

### 정책

- **저장 정책:** 비로그인 → 브라우저 저장 / 로그인 → 서버 저장

### 입력값 / 출력값

| 구분 | 내용 |
|------|------|
| 입력 | 유저의 상품 상세 페이지 조회 행동 |
| 출력 | 최근 본 상품 목록 표시 |

### 예외 케이스

| 예외 상황 | 처리 방식 | 상태 |
|-----------|-----------|------|
| 비로그인 → 로그인 전환 시 브라우저 목록 처리 | 정책 미결 | [TODO] |

### 우선순위

**P2** — 긴급/필수 표현 없음. 최대 개수·삭제 기능 모두 "일단", "급하진 않아요" 표현 사용

### 개발 참고사항

- 로그인/비로그인 상태에 따라 저장 위치가 달라지므로 기술팀 구현 방식 결정 필요
- 최대 표시 개수 미결 — 기술팀 결정 후 개발 착수

### QA 체크포인트

- [ ] 로그인 유저: 상품 조회 시 서버에 기록되고 목록에 표시되는가
- [ ] 비로그인 유저: 상품 조회 시 브라우저에 기록되고 목록에 표시되는가
- [ ] 비로그인 유저: 브라우저 재접속 후 목록이 유지되는가
- [ ] 로그인/비로그인 전환 시 처리 방식 — 정책 확정 후 테스트

## C02 · 구독 플랜 자체 변경

### 기능 정의

유저가 고객센터를 거치지 않고 직접 구독 플랜을 업그레이드 또는 다운그레이드할 수 있는 기능

### 기능 목적

운영팀이 처리하던 플랜 변경 요청을 유저 셀프서비스로 전환하여 운영팀 부담을 줄임

### 사용자 시나리오

**Happy Path (업그레이드)**
1. 유저가 플랜 변경 페이지에 접근
2. 상위 플랜 선택 후 업그레이드 요청
3. 즉시 상위 플랜으로 전환되고 유저에게 확인 안내

**Happy Path (다운그레이드)**
1. 유저가 플랜 변경 페이지에 접근
2. 하위 플랜 선택 후 다운그레이드 요청
3. 현재 구독 기간 만료 후 하위 플랜으로 전환 예약

**Happy Path (이력 확인)**
1. 유저가 플랜 변경 이력 페이지에 접근
2. 이전 플랜 변경 내역 목록 확인

**예외 시나리오**
- 업그레이드 시 차액 정산 방식 미결 → 결제 처리 흐름 설계 불가 (TODO)

### 기능 요구사항 (REQ)

| ID | 요구사항 | 출처 |
|----|---------|------|
| REQ-001 | 유저가 직접 플랜을 업그레이드할 수 있다 | 운영팀장 (CS 이메일) |
| REQ-002 | 유저가 직접 플랜을 다운그레이드할 수 있다 | 운영팀장 (CS 이메일) |
| REQ-003 | 업그레이드는 즉시 적용된다 | 운영팀장 (CS 이메일) |
| REQ-004 | 다운그레이드는 현재 구독 기간 만료 후 적용된다 | 운영팀장 (CS 이메일) |
| REQ-005 | 유저가 플랜 변경 이력을 확인할 수 있다 | 운영팀장 (CS 이메일) |

### 정책

- **구독 정책:** 업그레이드 즉시 적용 / 다운그레이드 기간 만료 후 적용
- **결제 정책:** 차액 정산 방식 — 미결 (내부 논의 필요)

### 입력값 / 출력값

| 구분 | 내용 |
|------|------|
| 입력 | 유저의 플랜 선택 및 변경 요청 |
| 출력 | 플랜 전환 처리 결과 / 변경 이력 표시 |

### 예외 케이스

| 예외 상황 | 처리 방식 | 상태 |
|-----------|-----------|------|
| 업그레이드 시 차액 정산 방식 미결 | 내부 논의 완료 후 결정 | [TODO] |

### 우선순위

**P2** — 운영팀 효율 개선 목적. 긴급/필수 표현 없음

### 개발 참고사항

- 차액 정산 방식이 결정되어야 업그레이드 결제 처리 로직 설계 가능
- 기술팀 결정 필요

### QA 체크포인트

- [ ] 다운그레이드 시 현재 기간 만료 후 전환 예약이 등록되는가
- [ ] 플랜 변경 이력이 유저에게 표시되는가
- [ ] 차액 정산 방식 — 정책 확정 후 결제 흐름 테스트

## C03 · 리뷰 신고 기능

### 기능 정의

각 리뷰에 신고 버튼을 제공하고, 신고된 리뷰를 기준에 따라 숨김 처리하는 기능

### 기능 목적

부적절한 리뷰를 신고하고 콘텐츠를 관리할 수 있도록 함

### 사용자 시나리오

**Happy Path (신고)**
1. 유저가 리뷰 목록에서 부적절한 리뷰를 발견
2. 해당 리뷰의 신고 버튼 클릭
3. 신고 접수 완료

**예외 시나리오**
- 신고 접수 후 숨김 처리 기준 — Conflict 미결 (PO 결정 필요)
- 허위 신고 악용 시 처리 방식 — 원문 언급 있으나 결론 없음 (TODO)

### 기능 요구사항 (REQ)

| ID | 요구사항 | 출처 |
|----|---------|------|
| REQ-001 | 각 리뷰에 신고 버튼을 제공한다 | 기획자 (슬랙 채널) |

> **REQ-002 (숨김 기준):** Conflict로 인해 확정 불가. PO 결정 후 추가.

### 정책

- **콘텐츠 정책:** 신고 후 숨김 처리 기준 — Conflict 미결

### 입력값 / 출력값

| 구분 | 내용 |
|------|------|
| 입력 | 유저의 신고 버튼 클릭 |
| 출력 | 신고 접수 완료 / 기준 충족 시 리뷰 숨김 처리 (기준 미결) |

### 예외 케이스

| 예외 상황 | 처리 방식 | 상태 |
|-----------|-----------|------|
| 신고 후 숨김 기준 | Conflict 미결 — PO 결정 필요 | [CONFLICT / TODO] |
| 허위 신고 악용 시 처리 | 정책 미결 | [TODO] |

### 우선순위

**P2** — 긴급 표현 없음

### 개발 참고사항

- 숨김 처리 기준(자동 숨김 vs 운영자 검토)이 결정되어야 신고 처리 로직 개발 착수 가능
- Conflict 미결 상태에서 개발 착수 불가

### QA 체크포인트

- [ ] 각 리뷰에 신고 버튼이 노출되는가
- [ ] 신고 버튼 클릭 시 신고가 정상 접수되는가
- [ ] 숨김 기준 충족 시 리뷰가 숨김 처리되는가 — **Conflict 해소 후 테스트 가능**

### Conflict

> **[CONFLICT] 리뷰 자동 숨김 기준**
>
> - **요구 A (기획자 + 창업자, 슬랙 채널):** 신고 3건 이상 시 자동 숨김 처리
> - **요구 B (운영팀, 주간 회의 메모):** 신고 1건이어도 운영자 검토 후 처리해야 함. 자동 숨김은 허위 신고 악용 가능성 있음
>
> → 개발 착수 전 Product Owner가 최종 정책을 결정해야 함
