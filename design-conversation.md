## 1. 결과물 임팩트 논의

### Q. output이 약하다는 고민 — 구체화 또는 대안 아이디어?

**현황 문제**: "PRD 자동 정리"가 1회성 문서로 끝나면 PM의 핵심 역량(모순 감지·의사결정 요청)이 드러나지 않음. 단순 받아쓰기처럼 보일 위험.

**브레인스토밍 결과 — 세 가지 방향 검토**

| 옵션 | 설명 | 결론 |
|------|------|------|
| A. 파싱 스크립트 | 구분자 split → 케이스별 PRD 자동 출력 | Standard 레벨로 포함 |
| B. 노션 자동 기입/협업 기능 | Notion API로 PRD 페이지 생성 + 버전 관리 | **제외 범위**로 결정 (Cowork 도전 분기로만, 협업/버전관리는 노션 ⑥항 제외) |
| C. 스펙 리스크 의사결정 요청서 | [TODO]·모순을 임팩트·긴급도로 우선순위화 | **Challenge 레벨**로 포함 (PM 역량 변별 핵심) |

→ 핵심 변별 포인트는 "[TODO]를 얼마나 놓치지 않고 잡아내는가"와 "모순을 단정하지 않고 의사결정으로 올리는가". 복합 케이스 2건에 의도적 모순을 심어 이 역량을 채점한다.

## 2. 레벨 구조 설계

**설계 원칙**: "1회성 PRD 받아쓰기" → "새 입력에도 도는 파이프라인" → "스펙 리스크 전략화"로 발전하는 누적형 구조 (PM 문서 구조화형 사다리).

### 최종 확정 레벨

```
🟢 Basic (100점)  — PM 수준
  → 15케이스를 표준 PRD 5섹션으로 구조화 + 모순 2건 감지 + [TODO] 표시
  → 제출: 노션 또는 구글 독스 공개 링크 (prd_output.md)

🟡 Standard (+30점, 누적 130점)  — 리드 PM 수준
  → Basic + 새 raw_requirements.txt에도 동일 품질로 도는 재현 파이프라인(/명령)
  → PRD 양식(섹션·우선순위 체계·[TODO] 표기)을 본인이 설계 + 개발·QA 관점 근거

🔴 Challenge (+30점, 누적 160점)  — 시니어 PM 수준
  → Standard + [TODO]·모순을 모은 "스펙 리스크 의사결정 요청서"
  → 리스크 등급(블로커/주의/참고) 기준·양식을 본인이 설계
```

## 3. 데이터 재설계 (구버전 → 신버전)

### Q. 기존 problem-06-secondboost 폴더와 노션 권위 스펙(Problem 06) 정합성 맞추기

**변경 전 (구버전 데이터 구조)**
```
data/input-01-easy.md   (슬랙 DM, 명확 케이스)
data/input-02-medium.md (슬랙 채널, 다수 관계자 모호)
data/input-03-hard.md   (CS이메일+슬랙 혼합, 모순 1건)
→ 3개 파일, 난이도 3단계, 모순 1건
```

**변경 후 (노션 ④ 스펙 기준)**
```
data/raw_requirements.txt  — 단일 평문 파일
구분자: `--- [케이스 N · 유형] ---`
총 15케이스: 슬랙 6 / CS이메일 4 / 회의메모 3 / 복합(모순) 2
```

**주요 변경 사항**
- 3개 .md 파일 → **단일 raw_requirements.txt** (노션 ④ 파일명·형식 일치)
- 난이도 3단계 분리 → **유형 4종 15케이스**로 재구성
- 모순 1건 → **모순 2건**(케이스 14·15)으로 확대 — [TODO] 변별력 강화
- 정답(유형 라벨·PRD) 미포함 — 학생이 직접 생성

### 의도적 데이터 품질 함정 (참여자가 처리해야 할 과제)

| # | 함정 | 위치 | 학생이 해야 할 처리 |
|---|------|------|---------------------|
| 1 | **모순 (동일 기능 상반 요구)** | 케이스 14: 매직링크 vs 비번+2FA / 케이스 15: 서버 vs 로컬 저장 | 본문 단정 금지 → [TODO] A안/B안 병기 후 의사결정 요청 |
| 2 | **구어체·이모지·줄임말** | 슬랙 6케이스 ("ㅎㅎ", "ㅇㅇ", "🙏", "ㅋㅋ") | 노이즈 제거, 요구사항만 추출 |
| 3 | **속기체 메모 (주어 생략·오탈자)** | 회의메모 11·12·13 ("결졍", "안남", 주어 생략) | 맥락 복원, 불확실분은 [TODO] |
| 4 | **불명확/미정 항목 다수** | 전 케이스 ("일단", "나중에", "알아서", "미정", "논의 중", "더 봐야할듯") | 추측 금지 → [TODO]로 분리, 약한 의견은 P3 |
| 5 | **약한 의견 vs 확정 요구 혼재** | 케이스 03 "있으면 좋고 없어도 되고", 케이스 07 "알아서 적절히" | 사실/의견 분리, 의견은 본문 단정 안 함 |

> 케이스 14는 **서로 다른 화자**(창업자 vs 보안 컨설턴트)의 채널 간 모순, 케이스 15는 **같은 화자**(창업자)가 한 스레드에서 번복하는 모순 — 두 종류의 모순 패턴을 모두 테스트한다.

## 4. Output 형식 결정

### Q. md 파일이면 안 됨 — 시각적으로 확인할 수 있어야 함

**결론**: 케이스당 5섹션 PRD 포맷(노션 ⑤)을 유지하되, 레벨별 HTML 샘플로 시각화.

| 파일 | 주요 시각 요소 |
|------|---------------|
| `output/sample-basic.html` | 유형 필터 버튼 + 15케이스 PRD 카드 + [TODO]/모순(CONFLICT) 하이라이트 + 유형 분포 통계 |
| `output/sample-standard.html` | 파이프라인 플로우 다이어그램 + PRD 양식 설계 표 + /generate 출력 샘플 렌더 |
| `output/sample-challenge.html` | 리스크 등급 프레임 + A안/B안 의사결정 카드 + 우선순위 점수 + 스펙 리스크 요약 테이블 |

> PRD 출력 포맷(케이스당 5섹션)은 노션 ⑤ 그대로: 기능 정의 / 사용자 시나리오(트리거→행동→결과) / 우선순위(P1/P2/P3+근거) / 예외 케이스 / 미결 항목(TODO).
